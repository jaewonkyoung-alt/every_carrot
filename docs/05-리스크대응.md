# 리스크 및 대응 방안

> 4주 안에 프로젝트를 완성할 때 흔히 겪는 문제와 해결책

---

## 리스크 1: Git 충돌 + 코드 꼬임

### 상황
- 같은 파일을 여러 명이 동시에 수정
- merge conflict 해결 과정에서 코드가 날아감
- "git push가 안 돼요" → force push → 다른 사람 코드 덮어씀

### 대응
| 예방 | 대처 |
|------|------|
| feature 브랜치 전략 사용 (각자 독립 브랜치) | 충돌 발생 시 고영민이 도와서 해결 |
| 매일 아침 `git pull origin develop` 습관화 | force push 절대 금지 (팀 규칙) |
| 프론트/백 폴더가 분리되어 있어 충돌 확률 낮음 | 모르면 push 전에 단톡방에 물어보기 |
| PR 머지 전 리뷰 1명 이상 | 최악의 경우 `git reflog`로 복구 가능 |

### Git이 처음인 팀원 안내
> 처음에는 `git add .` → `git commit` → `git push`만 반복하세요.
> 브랜치 전환이 헷갈리면 고영민한테 화면 공유하면서 같이 하세요.

## 리스크 2: 팀원 간 학습 속도 차이로 인한 진도 지연

### 상황
- Express 구조가 익숙하지 않아 API 하나 만드는 데 예상보다 시간 소요
- 에러 메시지를 혼자 해결하기 어려움
- 질문하기 부끄러워서 진도가 밀림

### 대응
| 예방 | 대처 |
|------|------|
| 1주차에 환경 세팅 + 기초 학습 시간 충분히 배정 | 2일 이상 막히면 페어 프로그래밍으로 전환 |
| 고영민이 라우터/컨트롤러 뼈대 작성 → 김동민은 SQL 쿼리와 응답 포맷 채우기 | 태스크 재분배 (더 작은 단위로 쪼개기) |
| 태스크를 API 1개 단위로 쪼개서 배분 | "모르면 바로 물어보기" 팀 문화 만들기 |
| 매일 15분 스탠드업 미팅 (진행 상황 공유) | 시딩/문서화 등 코딩 외 기여도 인정 |

### 태스크 단위 설계 원칙
```
❌ "채팅 기능 구현해 주세요"         → 범위가 너무 큼
✅ "GET /api/categories 만들어 주세요.
    categories 테이블에서 전체 목록을
    JSON으로 반환하면 됩니다."        → 명확하고 작음
```

## 리스크 3: Socket.io 채팅 구현 난이도

### 상황
- 실시간 통신 개념 자체가 생소
- 연결이 끊겼다 붙었다 하면서 디버깅 어려움
- 채팅방 구분, 메시지 순서 보장 등 예상 못한 이슈

### 대응
| 예방 | 대처 |
|------|------|
| 3주차에 배치 (1~2주차에 기초 다진 후) | 최소한의 기능만 구현 (입장/퇴장/메시지 송수신) |
| 고영민이 Socket.io 서버 담당 | 읽음 처리, 타이핑 표시 등은 과감히 스킵 |
| Socket.io 공식 튜토리얼 미리 읽어보기 | 최악의 경우: 폴링 방식으로 대체 (5초마다 GET) |
| 간단한 채팅 예제 먼저 따라 만들어보기 | 프론트는 UI만 먼저 만들고 나중에 연동 |

### 최소 구현 범위 (이것만 되면 OK)
```
1. 채팅방 입장 (join)
2. 텍스트 메시지 전송/수신
3. 메시지 DB 저장 (새로고침해도 유지)
4. 채팅방 목록 표시
```

## 리스크 4: 프론트-백 연동 지연

### 상황
- 백엔드 API가 안 끝나서 프론트가 놀고 있음
- API 응답 형식이 프론트 예상과 다름 (필드명 불일치 등)
- "내 쪽에선 되는데요?" → 서로 다른 데이터로 테스트

### 대응
| 예방 | 대처 |
|------|------|
| 1주차에 API 명세서 합의 (요청/응답 형식 확정) | API 변경 시 즉시 팀 공유 |
| 프론트는 Mock 데이터로 먼저 개발 | Postman 컬렉션 공유 (같은 데이터로 테스트) |
| API 명세를 docs/api-spec.md에 관리 | 2주차 중반부터 실제 연동 시작 |
| 응답 형식 통일: `{ success: true, data: {...} }` | 문제 생기면 화면 공유하면서 디버깅 |

### API 응답 형식 약속 (예시)
```json
// 성공
{
  "success": true,
  "data": { ... }
}

// 실패
{
  "success": false,
  "message": "에러 메시지"
}

// 목록 (페이지네이션)
{
  "success": true,
  "data": [ ... ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 45
  }
}
```

## 리스크 5: DB 스키마 변경

### 상황
- 개발 중에 "이 필드도 필요하네" → 테이블 수정
- 팀원마다 로컬 DB 스키마가 다름
- 외래키 관계 수정 시 기존 데이터 꼬임

### 대응
| 예방 | 대처 |
|------|------|
| 1주차에 ERD 확정 + 꼼꼼히 리뷰 | 스키마 변경 시 SQL 파일 업데이트 + 팀 공유 |
| `server/sql/init.sql`에 전체 스키마 관리 | 로컬 DB 리셋: `DROP → CREATE → init.sql 실행` |
| 초기 시딩 데이터도 SQL 파일로 관리 | 필드 추가는 `ALTER TABLE`로 문서화 |

### init.sql 예시
```sql
-- 기존 테이블 삭제 (개발 단계에서만)
DROP TABLE IF EXISTS messages;
DROP TABLE IF EXISTS chat_rooms;
DROP TABLE IF EXISTS products;
DROP TABLE IF EXISTS categories;
DROP TABLE IF EXISTS email_verifications;
DROP TABLE IF EXISTS users;

-- 테이블 생성
CREATE TABLE users ( ... );
CREATE TABLE categories ( ... );
-- ...

-- 시딩 데이터
INSERT INTO categories (name) VALUES
('전자기기'), ('도서'), ('의류/패션'), ('생활용품'), ('기타');
```

## 리스크 6: 시간 부족 (4주는 짧다)

### 상황
- 중간에 다른 과목 시험/과제와 겹침
- 예상보다 구현이 오래 걸림
- 마지막 주에 몰아치기

### 대응
| 예방 | 대처 |
|------|------|
| MVP에만 집중 (있으면 좋은 기능은 과감히 포기) | 3주차 말까지 핵심 기능 완성 목표 |
| 매일 30분이라도 코딩 (몰아치기 금지) | 4주차는 버그 수정 + 발표 준비에만 사용 |
| 매주 발표가 자연스러운 마일스톤 역할 | "70% 완성도로 시연 가능" 상태를 먼저 목표 |
| 어려운 기능(채팅)은 쉬운 버전 대비 | 최소한 CRUD + 채팅이 돌아가면 과제 통과 |

## 리스크 7: 팀 커뮤니케이션 부재

### 상황
- "그거 누가 하는 거였죠?"
- 같은 기능을 두 명이 만듦 / 아무도 안 만듦
- 카톡 읽씹, 회의 불참

### 대응
| 예방 | 대처 |
|------|------|
| 역할 분담 문서를 명확히 작성 | 주 2회 이상 짧은 미팅 (15분 스탠드업) |
| GitHub Issue로 태스크 관리 | 담당자 변경 시 즉시 공유 |
| 카톡/디스코드 팀 채널 활성화 | PM(경재원)이 매일 진행 상황 체크 |
| 중요한 결정은 문서로 남기기 | "약속 안 지키면 1/N 불이익" 팀 룰 수립 |

### 추천: 매일 스탠드업 (15분)
```
각자 3가지만 공유:
1. 어제 한 것
2. 오늘 할 것
3. 막히는 것
```
